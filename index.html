Version positioning

Pick the surface that matches the work style.

None of the plugins rely on background hooks or automatic writes. The difference is how much command surface, workflow state, and routing structure you want exposed during a Claude Code session.

How they work

Three operating models, drawn as workflows.

The family shares the same safety posture: keep one owner, split work only when it helps, validate with executable checks before semantic judgment, and treat memory or self-improvement as governed artifacts.

Based Claude flow

Explicit command chooses the mode.

Goal enters a commandplan, code, repair, validate, review, and more.
Owner keeps global statebased-developer coordinates scope, files, risks, and completion.
Specialists isolate workPlanner, scout, repairer, reviewer, validator, safety, memory, improver, or minimizer when useful.
Helpers produce evidenceQuality gates, traces, plans, memory cards, improvement proposals, and handoffs.
Review before trustChecks and fresh review come before durable learning or promotion.

Based Claudia flow

Artifacts are the workflow backbone.

Start creates state.based/state/current.md captures the current objective and next action.
Plan writes the path.based/plans/<slug>/plan.md becomes the resumable instruction spine.
Implement follows planThe owner works directly unless separation is useful.
Check and review record evidenceValidation and review notes land under .based/validation and .based/reviews.
Handoff or learnCompact state and memory drafts preserve what matters without raw transcript replay.

Based Claudista flow

The router chooses the lane.

Work command receives task/based-claudista:work is the normal entrypoint.
Router selects modeDirect work, planning, repair, validation, review, safety, memory, or improvement.
Control plane records state.based-claudista/ stores runs, traces, cards, improvements, handoffs, and plans.
Loop mode constrains autonomyManual, report-only, assisted-fix, or externally approved unattended mode.
Quality gate closes the runValidation evidence, reviewer separation, risks, and rollback notes keep outputs accountable.

Based Claudius flow

One router carries the whole family's contract.

Work command routes by phase/based-claudius:work moves through discover, plan, act, verify, review, handoff, and improve.
Owner keeps artifact-backed stateclaudius-developer writes to .claudius/ only on request.
Code-elegance gate checks the diffclaudius-slop-check.js makes the mechanical style pass checkable before review.
Memory and improvement stay gatedGoverned memory cards and proposal-driven improvement require validated evidence.
Review before trustDeterministic validation ladder ends with independent review, same as the rest of the family.

Shortest rule

Use Based Claude when you want breadth, Based Claudia when you want durable state first, Based Claudista when you want a router to choose the workflow lane, and Based Claudius when you want all of that merged into one flagship plugin.

Which one should be enabled?

Choose Based Claude for maximum coverage. It gives Claude a complete set of named workflows and specialist roles without turning on automatic background behavior.
Choose Based Claudia for transparent state. It is the cleanest fit when resumability and inspectable files matter more than a broad command catalog.
Choose Based Claudista for routed operations. It is the strongest fit for recurring work, controlled loop modes, and one-command task routing.
Choose Based Claudius for the consolidated flagship. It merges router, artifacts, reference depth, memory, and quality gate into one plugin instead of choosing between the three siblings.
